# 编码与解码本质

> 一句话定义：编码是把抽象信息映射为物理可存储/可传输的信号；解码是按同一套规则把信号还原为信息。

---

## 一、编码与解码的本质

**编码（Encode）：** 把抽象信息映射到离散符号（比特），使之可存储、可传输。

同一个信息在不同编码规则下，物理形态完全不同：

```
"你"（人类认知中的一个汉字）
   → UTF-8 规则  → [0xE4, 0xBD, 0xA0] (3 bytes)
   → UTF-16 规则 → [0x4F, 0x60]       (2 bytes)
   → GBK 规则    → [0xC4, 0xE3]       (2 bytes)
```

**编码的本质矛盾：** 信息本身只存在于语义层面，让计算机处理必须降维成离散符号，这个过程必然损失上下文——字节序列不携带"我是什么编码"的元信息。这正是乱码的根源。

**解码（Decode）：** 按同一套映射规则，把物理信号还原为抽象信息。

> 核心认知：编码和解码必须用同一套映射规则，否则还原出来的不是原始信息。类比——编码是把想法翻译成英文，解码是把英文翻译回想法。用英文写、用法文规则读，读出来的就是垃圾。不是信息丢了，是拿错了字典。

## 二、乱码的本质

**乱码不是数据损坏，而是用错误的解码规则去解读正确的编码数据。**

### 乱码产生的完整链路

```
原始信息: "你好"
   ↓ 编码（UTF-8）
字节序列: E4 BD A0 E5 A5 BD
   ↓ 存储/传输（字节完好无损）
   ↓ 解码（错用 GBK 规则来读！）
   ↓ GBK 把 E4 BD 解读为"浣"，A0 是半个不完整字 → "浣犲ソ"
显示: "浣犲ソ" ← 乱码
```

**字节一个没丢、一个没错。数据是完好的，错的是解读方式。**

### 三种不同性质的"乱码"

| 类型 | 原因 | 本质 |
|------|------|------|
| 编码-解码规则不匹配 | 用 UTF-8 编码，用 GBK 解码 | 字典用错了，数据完整，元信息丢失 |
| 字节序列物理损坏 | 网络丢字节、磁盘坏扇区翻 bit | 数据真正损坏，通常出现 `�` (U+FFFD) |
| 字体缺失 | 数据和解码都正确，但系统无对应字形 | 渲染链断了，显示为方块 `□` |

### 为什么中文世界乱码严重

历史遗留两套独立体系并存数十年：[[GBK|GB 系列]]（GB2312 → GBK → GB18030）与 [[Unicode]]（UTF-8/UTF-16）。英语世界几乎无此问题——[[ASCII]] 在所有编码体系的前 128 个码位兼容，`hello` 在 UTF-8、GBK、Latin-1 下字节完全相同。

## 三、Unicode 与 UTF-8 的关系

**一句话：[[Unicode]] 是"字典"（给全世界的字符编号），[[UTF-8]] 是"运输方案"（把这些编号存入字节序列）。**

- Unicode 分配码点（Code Point）给每个字符：`A` → U+0041，`你` → U+4F60，`😂` → U+1F602
- Unicode 不规定码点如何存到磁盘
- [[UTF-8]] 是变长编码方案，把 0~0x10FFFF 的码点映射为 1~4 字节

### UTF-8 编码规则

```
码点范围               UTF-8 编码                              字节数
U+0000-U+007F         0xxxxxxx                                 1
U+0080-U+07FF         110xxxxx 10xxxxxx                        2
U+0800-U+FFFF         1110xxxx 10xxxxxx 10xxxxxx               3
U+10000-U+10FFFF      11110xxx 10xxxxxx 10xxxxxx 10xxxxxx      4
```

示例："你"的码点 U+4F60 → UTF-8 为 `E4 BD A0`（3 字节）。

### 同一码点，不同运输方案

| 方案 | 特点 | `你`(U+4F60) 的字节 |
|------|------|---------------------|
| UTF-8 | 变长，ASCII 兼容，节省空间 | `E4 BD A0` (3 bytes) |
| [[UTF-16]] | 变长（2 或 4 bytes），BMP 字符定长 | `4F 60` (2 bytes) |
| UTF-32 | 定长 4 bytes，浪费空间但 O(1) 索引 | `00 00 4F 60` (4 bytes) |

> 你不能说"这个文件是 Unicode 编码的"——这句话技术上不精确，需要指明是 UTF-8 还是 UTF-16。

## 四、JS 为什么用 UTF-16，C++ 呢？

### JavaScript：被时代锁死

1995 年 JS 诞生时，Unicode 2.0 刚发布，码点还没超过 U+FFFF。[[UTF-16]] 看起来就是完美的定长编码——每个字符刚好 2 字节。

所以 JS 规范把"字符"定义为 16-bit 无符号整数，这导致了一个永恒的正确 bug：

```javascript
"你".length  // → 1（BMP 字符，2 bytes）
"😂".length  // → 2（emoji U+1F602 超出 BMP，UTF-16 需要代理对）
```

**本质：JS 不是"选择"了 UTF-16，而是"出生在 UTF-16 还是定长变量的年代"，被向后兼容性锁死了。**

### C++：不规定，让实现自己定

[[C++]] 诞生于 1985 年，当时 Unicode 还不存在。它从头就把字符串视为"字节序列"而非"字符序列"。

```cpp
char        // 至少 8 bits，编码不定
wchar_t     // 至少 16 bits，Windows 是 UTF-16，Linux 是 UTF-32
char16_t    // C++11，就是 UTF-16
char32_t    // C++11，就是 UTF-32
std::string // 只是 vector<char>，不保证任何编码
```

```cpp
std::string s = "你好";
s.length();  // UTF-8 下是 6（字节数），不是 2（字符数）！
```

### 对比

| | JS | C++ |
|---|---|---|
| 设计年代 | 1995 | 1985 |
| 字符串原子单位 | UTF-16 code unit (16 bit) | byte（不承诺） |
| `"你好".length` | 2 | UTF-8 下是 6 |
| 编码可变吗 | 几乎不可变 | 完全不规定 |
| 深层原因 | 被 Unicode 2.0 锁死 | 从一开始就把编码当用户问题 |

## 五、客户端（UTF-8）↔ 服务端（UTF-16）的转换链路

**转换无处不在，没有唯一转换点。**

```
客户端（JS，内存 UTF-16）
  │ ① 序列化：TextEncoder / JSON.stringify → UTF-8 字节流
  ▼
TCP 字节流（线路上是原始 bytes，无"字符集"概念）
  │ ② 传输：字节原封不动到达
  ▼
服务端（收包）
  │ ③ 反序列化：JSON parser 等库内部做 UTF-8 → 内存编码
  ▼
服务端（Java/C# 内存 UTF-16；Go/Rust/Python 3 内存 UTF-8）
```

**关键洞察：**

1. **网络线路上永远只有字节，没有字符集概念。** 编码转换从来不在 TCP/IP 层发生。
2. 转换发生在两个时刻：**序列化时**（内存→字节流）和**反序列化时**（字节流→内存）。
3. 中间层（Nginx、API Gateway）通常只透传字节，不转编码。
4. [[C++]] 的 `cout` 也不做转码——它只做 `write()` 系统调用，把字节原样搬到 [[stdout]]。这就是为什么输出到终端可能乱码：程序内部编码和终端预期编码不匹配。

## 六、GCC 的编码编译选项

[[GCC]] 通过三个独立选项控制编码：

| 选项 | 作用 | 默认值 |
|------|------|--------|
| `-finput-charset` | 告诉编译器源文件用什么编码 | UTF-8 |
| `-fexec-charset` | 二进制中字符串字面量的编码 | UTF-8 |
| `-fwide-exec-charset` | `wchar_t` 字面量（`L""`）的编码 | Linux 默认 UTF-32 |

### 数据流

```
源文件 .cpp
  ↓ -finput-charset 解码
编译器内部 Unicode 码点序列
  ↓ -fexec-charset 编码
可执行文件 .rodata 段
```

### C++11 的硬指定方式

```cpp
u8"你好"   // 强制 UTF-8  → char8_t[] (C++20)
u"你好"    // 强制 UTF-16 → char16_t[]
U"你好"    // 强制 UTF-32 → char32_t[]
L"你好"    // wchar_t，编码取决于 -fwide-exec-charset
```

> 用 `u8""`、`u""`、`U""` 前缀，编码写在代码里，不依赖编译选项。这是更稳健的做法。

### 运行时终端是否乱码？

终端乱码的判断公式：**程序内部的编码 vs 终端的预期编码 → 匹配则正常，不匹配则乱码。**

| 运行环境 | 终端默认编码 | `-fexec-charset=UTF-8` 会乱吗 |
|---------|------------|------------------------------|
| Linux (gnome-terminal 等) | UTF-8 | ✅ 不会 |
| macOS (Terminal/iTerm2) | UTF-8 | ✅ 不会 |
| Windows cmd.exe (中文) | GBK (936) | ❌ 会 |
| Windows Terminal + UTF-8 设置 | UTF-8 | ✅ 不会 |
| SSH 到远程 Linux | UTF-8 | ✅ 不会 |

**本质：乱码不发生在编译时，发生在运行时——程序的输出字节序列和终端期待的字节序列不是同一种"语言"。** [[GCC]] 只决定你输出什么语言，终端决定自己能听懂什么，[[C++]] 不会在中间帮你翻译。
