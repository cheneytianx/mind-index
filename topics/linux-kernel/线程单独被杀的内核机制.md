# 线程单独被杀的内核机制

> 一句话定义：应用层无法干净利落地杀单个线程，但 seccomp `KILL_THREAD` 能——因为它走了绕过信号框架的 `do_exit()` 内核特权路径。

## 背景

[[linux线程信号与进程级退出]] 已阐明：Linux 信号框架中所有致命信号都走 `do_group_exit()`，必然杀整个线程组。但如果内核安全机制判定某个线程越界，如何实现精准清除？

## 两种内核死刑路径

| | `do_exit()` | `do_group_exit()` |
|---|---|---|
| **触发者** | seccomp `KILL_THREAD` | seccomp `KILL_PROCESS` / 所有致命信号 |
| **范围** | 仅 `current` | 整个线程组 |
| **路径** | 绕过信号框架，直接执行退出 | 经过信号分发 → `zap_other_threads()` → 各线程各自 `do_exit()` |
| **共享资源** | 引用计数 -1，其他线程继续持有 | 所有线程退出后引用计数归零，资源释放 |
| **僵尸状态** | `EXIT_DEAD`（即刻可回收） | `EXIT_ZOMBIE`（等父进程 `wait()`） |
| **父进程** | 非最后线程：不通知；最后线程：通知 | 一定通知（进程消亡） |

核心实现：

```c
// kernel/seccomp.c
case SECCOMP_RET_KILL_THREAD:
    do_exit(SIGSYS);        // ← 直接进 do_exit，绕过信号框架

// kernel/exit.c
void __noreturn do_exit(long code)
{
    // 1. 设置 PF_EXITING，切断外部交互
    // 2. mm_struct / files_struct 引用计数 -1（共享的只减计数，不释放）
    // 3. exit_notify(): group_dead=false → EXIT_DEAD（静默回收）
    // 4. schedule() → 从运行队列移除，永不再调度
}
```

## 为什么用户态做不到单杀

**`complete_signal()` 强制了 SIGKILL 的全组退出：**

```c
static void complete_signal(int sig, struct task_struct *p, ...)
{
    if (sig == SIGKILL) {
        p->signal->flags |= SIGNAL_GROUP_EXIT;  // 写在线程组共享结构上
        zap_other_threads(p);                     // 给所有人发 SIGKILL
    }
}
```

用户态能调用的 kill 系列系统调用全部经过信号分发路径。即使用 `tgkill(tgid, tid, SIGKILL)` 精准命中目标线程，`complete_signal()` 依然会设置 `SIGNAL_GROUP_EXIT` 并唤醒所有其他线程。

| 用户态手段 | 能单杀？ | 原因 |
|---|---|---|
| `tgkill(tid, SIGKILL)` | ❌ | `complete_signal` 强制 group exit |
| `tgkill(tid, SIGTERM)` | ❌ | 默认 action = do_group_exit() |
| `tgkill(tid, SIGUSR1)` | ⚠️ | handler 注册则存活，没注册则杀全家 |
| `pthread_cancel(thread)` | ⚠️ | 异步 cancel 可终止但走用户态清理路径，不是内核级 do_exit |
| `pthread_exit()` | ✅ | 只能自杀，无法远程 |
| seccomp `KILL_THREAD` | ✅ | 内核 `do_exit()` 特权豁免 |

## 单线程死亡后的连锁反应

### 对其他线程

```
线程组: T1  T2  T3
         ↓
T2 触发 seccomp KILL_THREAD
         ↓
T2: do_exit()
    - PF_EXITING 标志 → 调度器移除
    - mm_struct 引用: 3→2（T1/T3 继续共享）
    - exit_notify(): group_dead=false → 不发信号给父进程
    - EXIT_DEAD → 静默回收
T1/T3: 继续正常运行，完全无感知
```

### 隐患一：锁未释放 → 死锁

```c
// T2 持锁时被干掉
pthread_mutex_lock(&global_lock);
// --- seccomp KILL_THREAD ---
// unlock 永远执行不到

// T1 → 死锁
pthread_mutex_lock(&global_lock);
```

解决：robust mutex。内核在 `do_exit()` → `exit_robust_list()` 遍历线程注册的 robust mutex 链表，标记 `owner = DEAD`，唤醒等待者返回 `EOWNERDEAD`。

### 隐患二：TLS 泄露

`__thread` 变量绑定的堆内存可能在析构函数执行前线程就死了，泄漏。

### 隐患三：主线程被杀的特殊性

Linux 不区分主/子线程，`thread_group_leader` 只是标志位。主线程被杀不会让进程死——其他线程继续，`/proc/pid/` 依然存在。但如果主线程是 `SIGINT` 的指定接收者，信号处理可能进入真空状态。

## 设计哲学

> Linux 在用户态**刻意不给线程级死刑能力**：进程是资源容器和信任边界，线程间无内存保护，部分线程死亡 = 地址空间状态不确定。内核不给这个能力，迫使应用自己管理线程生命周期。
>
> seccomp `KILL_THREAD` 是**裁判的特权豁免**：只有在内核安全机制判定线程行为越界时，才用 `do_exit()` 绕过信号框架做精准清除。这是一种**特权不对称**——能打破规则的只有裁判本人。
