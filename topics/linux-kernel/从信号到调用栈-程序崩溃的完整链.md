# 从信号到调用栈：程序崩溃的完整链

> 从底层信号机制出发，经过调试信息、信号 handler 的局限性，最终抵达 Breakpad 的崩溃转储与符号化方案——一条完整的"程序崩了之后发生了什么"链路。

---

## 一、程序崩了，底层发生了什么

### 1.1 硬件 → 内核 → 信号：三个层面的接力

程序执行 `*(int*)nullptr = 42` 时，CPU 的 MMU 发现地址 0 没有映射 → 触发 **page fault**（x86 上的 #PF 异常，异常号 14）。

CPU 的异常处理流程：

```
CPU 执行 *(int*)nullptr
  │
  ├─ MMU 查页表 → 地址 0 无效 → 触发 #PF
  │
  ├─ CPU: 保存现场（RIP/RSP/EFLAGS/错误码）→ 切换到内核栈 → 跳到 #PF handler 入口
  │
  └─ 内核 do_page_fault():
       ├─ 检查地址是否在进程的 VMA（虚拟内存区域）里?
       │   ├─ 不在 → MMU 无法修复的无效访问
       │   └─ bad_area()
       │        ├─ 检查是否可以回退（fixup section，用于内核自身的边界情况）
       │        └─ force_sig_fault(SIGSEGV, SEGV_MAPERR, address)
       │             │
       │             └─ 为当前线程设置 pending signal
       │                  在返回用户态之前，内核会调用 do_signal()
       │                  构造 sigframe 压到用户栈上
       │                  修改用户态 RIP 指向信号 handler
       │                  → iret 返回用户态
```

**关键点：** 内核不是"发消息"，而是在**返回用户态的路径上"劫持"执行流**，让线程先跑信号 handler。

### 1.2 同步信号 vs 异步信号

| | 同步信号 | 异步信号 |
|---|---|---|
| 触发者 | 当前线程自己的指令（SIGSEGV/SIGFPE/SIGILL/SIGBUS/SIGTRAP）| 其它进程或内核（SIGKILL/SIGTERM/SIGABRT 来自 kill）|
| 送达确定性 | 当前线程必然收到 | 任意线程都可能收到 |
| 上下文可靠性 | ucontext 精确指向故障指令 | ucontext 指向"被打断的任意一条指令" |
| 重新触发 | 返回 handler 后 CPU 自动重新执行故障指令 | 不自动——需要用 `tgkill` 重新发给自己 |

对 Breakpad 来说，**同步信号的处理更简单更可靠**——崩溃现场就是故障指令的"犯罪现场"。

## 二、应用层的信号处理可以干什么

### 2.1 信号处理的两级 API

```c
// 旧式：只拿到信号编号
signal(SIGSEGV, handler);       // void handler(int sig);

// 新式：拿到完整上下文
struct sigaction sa;
sa.sa_sigaction = handler;      // void handler(int sig, siginfo_t* info, void* ucontext);
sa.sa_flags = SA_SIGINFO;
sigaction(SIGSEGV, &sa, NULL);
```

`siginfo_t` 提供：
- `si_signo`：信号编号
- `si_code`：信号码（SEGV_MAPERR=访问了没映射的地址, SEGV_ACCERR=访问了没权限的地址）
- `si_addr`：触发故障的内存地址

`ucontext_t` 提供崩溃时刻的完整 CPU 寄存器快照，以及 FP/SIMD 寄存器状态。

### 2.2 信号 handler 能做的和不能做的

**能做的事（async-signal-safe）：**

| 操作 | 函数 | 说明 |
|------|------|------|
| 写原始字节到 fd | `write(2)` | 最基础的 I/O |
| 原子更新全局变量 | `volatile sig_atomic_t` | 设置标志位 |
| 发信号 | `kill(2)`, `tgkill(2)` | 通知其它进程 |
| fork 子进程 | `fork(2)` | 子进程不受信号上下文限制 |
| 直接退出 | `_exit(2)`, `_Exit(2)` | 跳过 atexit 清理 |

**不能做的事（不可重入，会死锁或二次崩溃）：**

| 操作 | 为什么危险 |
|------|-----------|
| `malloc`/`free` | 可能正持堆锁 → 死锁 |
| `printf`/`fprintf` | 内部加锁 FILE* → 死锁 |
| `backtrace()` | glibc 实现内部调 `malloc` |
| `dladdr()` | 动态链接器内部加锁 |
| C++ 异常 | 依赖 malloc + 栈展开 |

详见 `man 7 signal-safety`。

### 2.3 信号栈：防止雪上加霜

如果崩溃本身就是栈溢出导致的（比如递归炸栈），handler 再在原栈上执行 → 栈指针指向非法区域 → 二次崩溃 → 进程直接被杀且无法生成 dump。

```c
// 分配独立信号栈
stack_t ss;
ss.ss_sp = malloc(SIGSTKSZ);    // 8192 字节起步，实际建议至少 16KB
ss.ss_size = SIGSTKSZ;
ss.ss_flags = 0;
sigaltstack(&ss, NULL);

// 注册时加 SA_ONSTACK
sa.sa_flags |= SA_ONSTACK;
```

[[linux线程信号与进程级退出]] 里讨论过信号与线程的交互，所有线程共享信号 disposition 但可以有自己的信号掩码。

## 三、为什么信号 handler 里不能做复杂逻辑：两种"栈坏了"

### 3.1 字面意义：栈内存被破坏

栈缓冲区溢出 → 覆盖返回地址/帧指针/canary → SIGSEGV 发生时栈上的内容已不可信。

内核 deliver 信号时在栈上压 sigframe——如果栈指针已被破坏为非法值 → 连压栈都失败 → 进程静默被杀（handler 不被调用）。这就是 Breakpad 使用 `SA_ONSTACK` 的原因。

### 3.2 更致命的：执行上下文不可信

即使栈是完好的（比如只是解引用 NULL），handler 运行时的程序状态也是**不可知的**：

```c
void crash_handler(int sig) {
    // 这时候程序之前在做什么？
    // 它可能在 malloc 里，正持有一个 arena lock
    // 它可能在 printf 里，正持有一个 FILE* 的锁
    // 它可能在遍历一个链表，指针刚更新了一半
}
```

信号是**异步到达的**，它可以打断任何一条指令。你不知道被打断的函数是否持有某个全局锁——所以你不能在 handler 里做任何可能触及同一个锁的操作。

**`fork()` 是少有的逃脱路径：** handler 里 `fork()` 是 signal-safe 的。子进程有完整的地址空间拷贝，所有锁状态被冻结——`malloc` 的锁可能被持有，但子进程不会再有另一个线程去碰它。所以在子进程里可以安全地调用 `backtrace()`、写文件、甚至 `exec()` 一个外部工具。

## 四、调试信息到底是什么

### 4.1 本质：编译器丢弃的语义映射

编译过程将人类可读的源码翻译为机器码，大量语义被丢弃：

```
源代码                                  二进制
------                                  ------
int count = 0;                          8b 45 f8
count += items[i];                      01 45 f8
return count;                           8b 45 f8 c9 c3
```

**调试信息就是那张被丢弃的"翻译对照表"**——记录从机器码回到源码世界的所有映射关系。

### 4.2 DWARF 的核心组成

| 段 | 内容 | 用途 |
|----|------|------|
| `.debug_info` | 类型、变量、函数描述（核心）| DIE 树结构，描述一切编译单元内的符号 |
| `.debug_line` | 行号映射表 | "地址 0x4011a0 ↔ main.c 第 42 行" |
| `.debug_frame` | 调用帧信息 (CFI) | GDB 里 `bt` 能展开栈帧的元数据 |
| `.debug_abbrev` | 缩写表 | 压缩 .debug_info 中的重复元数据 |
| `.debug_str` | 字符串表 | 函数名、文件名、类型名 |

CFI 的工作原理：对每个函数，记录一条规则——"这个位置的返回地址存在 rbp+8，上一个帧指针存在 rbp"。这样即使栈帧格式不标准（比如 `-fomit-frame-pointer`），调试器也能逐帧回溯。

### 4.3 DWARF 与 BTF 的关系

[[BTF (BPF Type Format) 详解|BTF]] 是 DWARF 的极度精简子集，由 `pahole` 工具转换生成。它只保留 eBPF 需要的类型布局信息，大小比 DWARF 小 100-1000 倍，可以嵌入内核镜像。而 DWARF 必须外部存储（`-dbg` 包、`debuginfod` 服务）。

### 4.4 分离式调试信息

```bash
# 提取
objcopy --only-keep-debug myapp myapp.debug
# 剥离
strip --strip-debug myapp
# 建立关联
objcopy --add-gnu-debuglink=myapp.debug myapp
```

生产环境部署剥离后的轻量二进制，调试时通过 `debuginfod` 服务按 build-id 自动下载对应符号。Breakpad 的 `dump_syms` 就是对这些 DWARF 数据的再提取和格式转换。

## 五、Breakpad 概述与底层原理

### 5.1 三件套

```
client library（嵌入程序）→ 崩溃时生成 minidump
dump_syms              → 从带 DWARF 的二进制提取符号文件
minidump_stackwalk     → 服务端符号化，把地址还原为人类可读调用栈
```

### 5.2 核心架构：clone 子进程 + ptrace

这是 Breakpad 对"信号 handler 不能做复杂逻辑"这个约束的精妙解法。

```
崩溃线程（信号 handler，跑在独立信号栈上）
    │
    ├─ SignalHandler(sig, siginfo, ucontext)
    │    │
    │    ├─ 保存 siginfo + ucontext 到全局变量 g_crash_context_
    │    │
    │    └─ HandleSignal() → GenerateDump()
    │         │
    │         ├─ sys_clone(ThreadEntry, stack, CLONE_FS|CLONE_UNTRACED)
    │         │   克隆子进程，共享地址空间但不共享线程
    │         │
    │         ├─ sys_prctl(PR_SET_PTRACER, child_pid)     允许子进程 ptrace 自己
    │         ├─ pipe write 'a' → 通知子进程可以开始了
    │         └─ sys_waitpid(child_pid)                   等待子进程完成 dump
    │
    └─ 恢复信号默认处理 → 信号重新触发 → 进程终止

克隆出的子进程（正常上下文，无信号限制）
    │
    ├─ ThreadEntry()
    │    ├─ 阻塞在管道上等待父进程 Ready 信号
    │    └─ DoDump()
    │         │
    │         └─ WriteMinidump()
    │              ├─ ptrace(PTRACE_ATTACH, 每个线程)    挂起所有线程
    │              ├─ ptrace(PTRACE_GETREGS, tid)        读取每个线程寄存器
    │              ├─ ptrace(PTRACE_PEEKDATA, tid, addr) 读取栈内存
    │              ├─ 读 /proc/<pid>/maps                 获取内存映射
    │              ├─ 读 /proc/<pid>/auxv                 获取 ELF 辅助向量
    │              ├─ 构造 minidump 文件（无需 libc，使用自定义实现）
    │              └─ sys_exit(0)
```

**三个设计要点：**

1. **CLONE_FS | CLONE_UNTRACED**：`CLONE_FS` 让子进程看到的 `/proc/<parent_pid>/maps` 就是父进程崩溃瞬间的内存布局。`CLONE_UNTRACED` 防止子进程继承 ptrace 附加关系。
2. **崩溃线程用 ucontext 而非 ptrace**：ptrace 读回来的是"线程停在 handler 里"的寄存器，不是崩溃瞬间。ucontext 是内核 deliver 信号时保存的原始快照。
3. **自定义 libc 实现**：Breakpad 在 `common/linux/linux_libc_support.h` 和 `third_party/lss/linux_syscall_support.h` 中提供了独立的 `memset`/`memcpy`/syscall 直调，确保子进程中即使 libc 状态被破坏，dump 功能也不受影响。

## 六、崩溃时 Breakpad 做了什么 + Minidump 详解

### 6.1 信号注册

```cpp
// exception_handler.cc
const int kExceptionSignals[] = {
  SIGSEGV, SIGABRT, SIGFPE, SIGILL, SIGBUS, SIGTRAP
};
sa.sa_flags = SA_ONSTACK | SA_SIGINFO;   // 独立信号栈 + 完整上下文
sa.sa_sigaction = SignalHandler;
```

安装前保存旧 handler，卸载时恢复——不破坏其它库注册的 handler。

### 6.2 崩溃时的精确执行流

```
SignalHandler(sig, siginfo, ucontext)
  │
  ├─ first_chance_handler → V8 Wasm 可恢复信号给引擎先处理
  │
  ├─ 遍历 g_handler_stack_，找注册的 ExceptionHandler
  │
  ├─ HandleSignal()
  │    ├─ filter_ 回调 → 用户可拒绝本次 dump
  │    ├─ sys_prctl(PR_SET_DUMPABLE, 1)
  │    ├─ 填充 g_crash_context_ = {siginfo, tid, ucontext, float_state}
  │    ├─ crash_handler_ 回调 → 用户可选额外处理
  │    └─ GenerateDump(&g_crash_context_)
  │         ├─ 分配子进程栈 (16384 字节)
  │         ├─ sys_clone(ThreadEntry, ...)
  │         ├─ sys_prctl(PR_SET_PTRACER, child_pid)
  │         ├─ pipe write → 子进程开始
  │         └─ sys_waitpid(child_pid)
  │
  ├─ InstallDefaultHandler(sig)  恢复 SIG_DFL
  └─ tgkill(getpid(), tid, sig)  重新发给自己 → 触发默认处理 → core dump(可选)
```

### 6.3 Minidump 文件格式

Minidump 是 Microsoft 定义的崩溃转储格式（Breakpad 在 C99 上重实现了全部结构，见 `minidump_format.h`）。文件结构为 header + stream 数组：

```
MDRawHeader (签名 "PMDM" | 时间戳 | stream_count | stream_directory_rva)
  │
  ├─ Stream ①  ThreadListStream         每个线程的寄存器+栈
  ├─ Stream ②  ModuleListStream         已加载模块列表 (.so/.exe)
  ├─ Stream ③  MemoryListStream          收集到的内存区域汇总
  ├─ Stream ④  ExceptionStream           异常类型/地址/崩溃线程CPU上下文
  ├─ Stream ⑤  SystemInfoStream          OS/CPU 信息
  ├─ Stream ⑥  /proc/cpuinfo             原样
  ├─ Stream ⑦  /proc/<pid>/status        进程状态
  ├─ Stream ⑧  /etc/lsb-release          发行版信息
  ├─ Stream ⑨  /proc/<pid>/cmdline       启动命令行
  ├─ Stream ⑩  /proc/<pid>/environ       环境变量
  ├─ Stream ⑪  /proc/<pid>/auxv          ELF 辅助向量
  ├─ Stream ⑫  /proc/<pid>/maps          完整内存映射（原样）
  └─ Stream ⑬  DSODebugStream            所有已加载 .so 的 link_map 链表
```

### 6.4 每条信息的来源

| Stream | 内容 | 数据来源 |
|--------|------|---------|
| **Thread List** — 崩溃线程 | 崩溃时刻的寄存器 (IP/SP/BP/通用寄存器/浮点) | 内核 deliver 信号时保存的 **`ucontext_t`**（handler 的第三个参数）|
| **Thread List** — 崩溃线程栈 | 栈内存内容 | `ptrace(PTRACE_PEEKDATA)` 从 ucontext 记录的 SP 读取 |
| **Thread List** — 其他线程 | 寄存器 + 栈 | `ptrace(PTRACE_GETREGS)` + `ptrace(PTRACE_PEEKDATA)` |
| **Module List** | .so 文件名 / 基址 / 大小 / **ELF build-id** | `/proc/<pid>/maps` 解析 + ptrace 读 ELF header → `.note.gnu.build-id` |
| **Exception** | 信号编号 / 信号码 / 故障地址 / 额外信息 | `siginfo_t`（内核填充）→ `si_signo` / `si_code` / `si_addr` |
| **SystemInfo** | OS 版本 / CPU 型号 / 核心数 | `uname(2)` + `/proc/cpuinfo` |
| **DSO Debug** | 所有已加载 .so 的 **link_map 链表** | `auxv[AT_PHDR]` → 遍历 program headers → 找到 `PT_DYNAMIC` → 读 `.dynamic` → `DT_DEBUG` → ptrace 读 `r_debug` → 遍历 `link_map` 链表 |

**为什么 Module List 需要 build-id：** 这是服务端匹配符号文件的唯一凭证。同一个 libc.so 在两个不同系统上即使文件名一样但版本不同，build-id 也不同。按 build-id 匹配保证符号和二进制严格对应——这是 Breakpad / Crashpad 符号化正确性的根基。

**为什么 DSO Debug 需要 link_map：** `/proc/<pid>/maps` 只给文件路径和地址范围，不保证这些 .so 是通过动态链接器加载的（可能是 `dlopen` 的），也不保证文件路径是规范化的。link_map 是 ld.so 内部唯一权威的已加载库链表，和 `maps` 交叉验证。

### 6.5 模块过滤逻辑

```cpp
// 只保留有文件名、offset==0或executable、size>=4096 的映射
// 过滤掉 [vdso] / [stack] / 匿名 mmap
static bool ShouldIncludeMapping(const MappingInfo& mapping) {
    if (mapping.name[0] == 0 ||
        (mapping.offset != 0 && !mapping.exec) ||
        mapping.size < 4096)
        return false;
    return true;
}
```

## 七、Demo：使用 Breakpad 进行崩溃堆栈分析

### 7.1 最小程序

```cpp
// crash_demo.cpp
#include <iostream>
#include "client/linux/handler/exception_handler.h"

using namespace google_breakpad;

bool MinidumpCallback(const MinidumpDescriptor& descriptor,
                      void* context, bool succeeded) {
    printf("%s: %s\n", succeeded ? "✅" : "❌", descriptor.path());
    return succeeded;
}

bool FilterCallback(void* context) {
    printf("💥 Crash detected, generating minidump...\n");
    return true;
}

void CauseSegfault() {
    volatile int* p = nullptr;
    *p = 42;  // ← SIGSEGV here
}

int main() {
    MinidumpDescriptor descriptor("/tmp/crashes");
    ExceptionHandler handler(descriptor, FilterCallback,
                             MinidumpCallback, nullptr,
                             true,  // install_handler
                             -1);   // server_fd: 进程内模式
    CauseSegfault();
    return 0;
}
```

### 7.2 编译

```bash
# 1. 编译 Breakpad
cd /tmp/breakpad && ./configure && make -j$(nproc)

# 2. 编译 demo（带 -g 生成 DWARF 调试信息）
g++ -g -I/tmp/breakpad/src \
    -o crash_demo crash_demo.cpp \
    -L/tmp/breakpad/src/client/linux -lbreakpad_client \
    -lpthread -lstdc++
```

### 7.3 运行 → 符号化 → 得到调用栈

```bash
# 运行触发崩溃
mkdir -p /tmp/crashes && ./crash_demo
# → /tmp/crashes/<guid>.dmp

# 提取符号
/tmp/breakpad/src/tools/linux/dump_syms/dump_syms ./crash_demo > crash_demo.sym

# 符号文件必须放在按模块名+build-id组织的目录结构中
# dump_syms 输出第一行: MODULE Linux x86_64 <BUILD_ID> crash_demo
# 需要: ./symbols/crash_demo/<BUILD_ID>/crash_demo.sym
# （工具里有脚本自动做这个：src/tools/linux/symupload/）

# 符号化：把地址变成函数名+行号
/tmp/breakpad/src/processor/minidump_stackwalk \
    /tmp/crashes/<guid>.dmp ./symbols 2>/dev/null
```

### 7.5 工程决策清单

| 决策点 | 选项 | 适用场景 |
|--------|------|---------|
| dump 生成 | 进程内 (`server_fd=-1`) vs 进程外 | 简单场景用进程内；多进程/沙箱环境用进程外 |
| 写入目标 | 文件路径 vs fd | 路径自动命名；fd 配合管道异步上传不落盘 |
| 符号管理 | 本地 `dump_syms` vs Symbol Server | 小团队本地；生产需 build-id 索引的符号服务器 |
| 沙箱兼容 | 白名单 syscall | 需要允许 `ptrace`/`clone`/`pipe`/`write` |
| 信号栈 | 默认 16KB vs 调整 | 栈帧极深（大量递归/协程）需增大 |

### 7.6 主控 dump 和自定义数据

```cpp
// 主动生成（不依赖崩溃）
handler.WriteMinidump();

// 注册额外内存区域（JIT 代码段等）
int important_state[1024];
handler.RegisterAppMemory(important_state, sizeof(important_state));

// 注册自定义内存映射
handler.AddMappingInfo("jit-code", build_id, base_addr, size, offset);
```

## 全链路总结

```
源码 .cpp
  │  gcc -g → DWARF 嵌入
  ▼
二进制 + 调试信息
  │  dump_syms → 提取 build-id + 函数名/文件/行号映射
  ▼
符号文件 (.sym)
  │
  │                        ┌─────────────────────────────────┐
  │                        │  运行时                          │
  │                        │                                 │
  │                        │  CPU 执行非法指令                │
  │                        │    → 硬件异常                    │
  │                        │    → 内核构造 siginfo+ucontext   │
  │                        │    → 信号 handler (Breakpad)     │
  │                        │      → 保存 g_crash_context_     │
  │                        │      → clone 子进程              │
  │                        │        → ptrace 读内存/寄存器    │
  │                        │        → 写 minidump (.dmp)      │
  │                        │        → 子进程退出              │
  │                        │    → 恢复 SIG_DFL                │
  │                        │    → 信号重新触发               │
  │                        │    → 进程终止                    │
  │                        └─────────────────────────────────┘
  │
  │  minidump_stackwalk(.dmp + .sym)
  ▼
人类可读调用栈
  "crash_demo!CauseSegfault at crash_demo.cpp:23"
```

从 CPU #PF 到"crash_demo.cpp:23"——这一整条链路上的每一个环节都不可跳过：没有信号机制就没有崩溃的可拦截性、没有 ucontext 就没有精确崩溃现场、没有 clone+ptrace 就没有安全的 dump 生成、没有 DWARF 就没有地址到源码的映射、没有 build-id 就没有跨版本符号匹配的正确性保证。
