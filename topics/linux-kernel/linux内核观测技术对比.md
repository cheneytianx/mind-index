# Linux 内核观测与拦截技术全景对比

> 从统一的维度框架出发，对比 eBPF、内核模块、LD_PRELOAD、inotify/fanotify、audit、libpcap、netlink 七项技术的本质差异与工程取舍。

---

## 一、建立对比维度

评价任何观测/拦截技术，先从这几个维度看：

| 维度 | 含义 |
|------|------|
| 观测层级 | 你能看到什么？syscall？网络包？文件变更？内核内部函数？ |
| 侵入性 | 对被观测系统的性能、稳定性、行为影响程度 |
| 安全边界 | 出 bug 能不能搞挂内核 / 影响其他进程 |
| 部署成本 | 需要改代码、重编内核、重启进程吗？ |
| 粒度与灵活性 | 能否按条件过滤？能否写自定义逻辑？ |
| 覆盖范围 | 单进程还是全系统？ |
| 实时性 | 延迟是纳秒级还是毫秒级？ |

---

## 二、分类框架

这七项技术先划归到三个大类：

**主动注入代码型** — 你把代码塞到目标执行空间，决定观测什么、怎么处理：
- [[eBPF]]：内核沙箱执行，verifier 保安全
- [[内核模块|LKM]]：无限制内核代码，能力无上限但风险无上限
- [[LD_PRELOAD]]：用户态目标进程内注入

**被动订阅事件型** — 内核预定义事件，你选感兴趣的订阅：
- [[inotify]] / [[fanotify]]：VFS 文件系统事件
- [[audit]]：syscall 级审计框架，合规导向
- [[netlink]]：内核子系统状态变更的消息总线

**旁路复制数据型**：
- [[libpcap]]：网络包旁路复制，pcap 生态

## 三、逐项深度分析

### eBPF

```
层级：内核态任意挂载点（syscall、kprobe、tracepoint、XDP、cgroup、LSM）
机制：JIT 编译的沙箱字节码，挂载到内核 hook 点
```

- **观测范围**：全系统、全栈——网卡收包到 syscall 到调度器，几乎无死角
- **侵入性**：极低。不改被观测进程，不改内核源码。热加载/热卸载
- **安全性**：verifier 保证不会死循环、不会越界访问、不会 panic
- **性能**：纳秒级。JIT native code，per-CPU map 无锁
- **灵活性**：最高。可以写任意过滤逻辑、聚合统计、频率采样
- **部署**：不需要重启。内核 4.x 基本可用，5.x+ 才好用
- **局限**：指令数限制（100万）、不能调用任意内核函数、调试困难

**一句话定位：** 可编程的内核级观测平面，安全性和灵活性的最佳平衡点。

---

### 内核模块（LKM）

```
层级：内核态，无限制
机制：动态加载的内核代码，与内核同权限
```

- **观测范围**：无限制。能做 eBPF 能做的一切，还能做 eBPF 做不了的
- **侵入性**：高。本身就是内核的一部分
- **安全性**：零保护。一个 bug 就是 kernel panic 或数据损坏
- **性能**：理论最高——没有 verifier 限制、没有指令数上限
- **灵活性**：最高。可以 hook 任意函数、修改任何行为
- **部署**：需要编译（依赖内核头文件）、insmod 加载。签名内核需签名模块
- **局限**：维护成本极高、内核版本耦合、安全风险不可控

**eBPF 替代不了内核模块的场景：**
- 需要修改内核行为（不只是观测）
- 需要长时间持有锁或睡眠等待
- eBPF 指令数/复杂度不够的超复杂逻辑

---

### LD_PRELOAD

```
层级：用户态，libc/动态库函数层
机制：动态链接器拦截，替换共享库符号
```

- **观测范围**：仅目标进程，仅动态链接的函数调用
- **侵入性**：中。不改源码但改变了进程的运行时行为
- **安全性**：崩溃只影响目标进程，不影响内核
- **性能**：函数调用级开销（纳秒），但每次调用都经过 wrapper
- **灵活性**：中。只能拦截动态链接的符号，Go/Rust 静态链接完全无效
- **部署**：需要重启目标进程设环境变量，或 ld.so.preload 全局
- **局限**：**绕过极易** — 直接 syscall() 绕过 libc；setuid 程序忽略

**典型用途：** libfaketime、jemalloc 替换、I/O 追踪

---

### inotify / fanotify

```
层级：VFS 层，文件系统事件
机制：内核 fsnotify 子系统，通过 fd 向用户态推送事件
```

| 特性 | inotify | fanotify |
|------|---------|----------|
| 观测范围 | 单个目录/文件（不递归） | 整个挂载点/文件系统 |
| 事件类型 | open/close/modify/create/delete | 同上 + 访问决策（allow/deny） |
| 信息量 | 文件名+事件类型。**无 pid** | 有 pid、fd，可读文件内容 |
| 性能 | 高频事件风暴可能丢队列 | 拦截模式同步等待用户态决策，代价大 |
| 部署 | 无需特权 | 拦截模式需 CAP_SYS_ADMIN |

**fanotify 的杀手级能力：** 拦截模式可以在进程 open 文件前决定是否允许——杀毒软件就是这么做的。

---

### audit（Linux Audit Subsystem）

```
层级：syscall 层 + 部分内核事件
机制：内核 audit 子系统 → auditd 守护进程 → 日志
```

- **观测范围**：所有 syscall（可按规则过滤）、文件访问、网络连接、用户认证
- **性能**：**相对较差**。每条记录格式化写日志，高并发下成瓶颈
- **信息量**：丰富：syscall 号、参数、返回值、pid、完整路径
- **灵活性**：规则式过滤（`-a always,exit -F arch=b64 -S open`），不能做复杂逻辑
- **部署**：内核内建，auditctl 添加规则即可
- **局限**：性能差、不能聚合/采样、不能修改行为

**与 eBPF 的关键差异：**
- audit 是日志导向（事后分析），eBPF 是实时可编程（当场聚合/过滤/响应）
- audit 无法做"只统计每秒 write 次数"这种聚合

---

### libpcap（BPF classic / AF_PACKET）

```
层级：网络栈，数据链路层
机制：AF_PACKET socket + 经典 BPF 过滤器 → mmap ring buffer
```

- **观测范围**：纯网络包。二层及以上，原始帧/包
- **性能**：每个包复制到用户态，10Gbps+ 容易丢包
- **信息量**：完整包内容，但 **不知道是哪个进程收发的**
- **部署**：CAP_NET_RAW 或 root
- **局限**：只看网络、不关联进程、加密流量看到密文

**vs eBPF XDP/TC 的网络观测：**
- libpcap 在协议栈出口/入口复制，XDP 在网卡驱动层处理
- eBPF 可以关联 socket 和进程，libpcap 做不到
- eBPF 可以在内核态聚合，libpcap 必须全量抓到用户态

---

### Netlink

```
层级：内核态 → 用户态的通信通道
机制：特殊的 socket 族（AF_NETLINK），内核主动推送事件
```

- **观测范围**：内核子系统事件通知：路由、网卡、进程事件（[[proc connector|CONNECTOR]]）、SELinux
- **侵入性**：零。纯被动接收
- **性能**：高。内核批量推送，用户态按需消费
- **灵活性**：低。只能订阅预定义事件类型，不能自定义
- **局限**：不能自定义观测点，只能用内核提供的接口

**典型 Netlink 协议族：**
- `NETLINK_ROUTE` — 路由/网卡/地址变更
- `NETLINK_CONNECTOR` — 进程 fork/exec/exit/uid 事件
- `NETLINK_AUDIT` — audit 事件传输通道
- `NETLINK_KOBJECT_UEVENT` — 设备热插拔（udev 依赖）
- `NETLINK_NETFILTER` — conntrack/nflog 事件

## 四、全景对比矩阵

| 技术 | 观测层 | 全系统 | 可编程 | 安全 | 性能影响 | 能否拦截 | 无需重启 |
|------|--------|--------|--------|------|----------|----------|----------|
| eBPF | 内核全栈 | ✅ | ✅ 高 | ✅ verifier | 极低 | ⚠️ 有限 | ✅ |
| 内核模块 | 内核全栈 | ✅ | ✅ 无限 | ❌ | 取决于实现 | ✅ | ✅ |
| LD_PRELOAD | 用户态 libc | ❌ 单进程 | ✅ 中 | ✅ 进程隔离 | 低 | ✅ | ❌ 需重启 |
| inotify/fanotify | VFS 文件 | ✅ | ❌ 规则式 | ✅ | 低~中 | ✅ (fanotify) | ✅ |
| audit | syscall | ✅ | ❌ 规则式 | ✅ | 中~高 | ❌ | ✅ |
| libpcap | 网络包 | ✅ | ❌ cBPF | ✅ | 中 | ❌ | ✅ |
| netlink | 内核事件总线 | ✅ | ❌ 订阅式 | ✅ | 极低 | ❌ | ✅ |

## 五、工程场景的最佳选择

| 场景 | 首选 | 备选 | 原因 |
|------|------|------|------|
| 容器全量文件/网络审计 | **eBPF** | audit | eBPF 轻量可编程，audit 太重 |
| 实时阻断高危命令 | **LD_PRELOAD** | eBPF LSM | 部署轻量，crash 影响面小 |
| 内核级文件拦截（杀毒） | **fanotify** | eBPF LSM | API 成熟，同步决策路径清晰 |
| 合规审计日志（PCI-DSS） | **audit** | eBPF | 标准认可，生态成熟 |
| 文件变更监听（配置管理） | **inotify** | fanotify | 最轻量，无特权需求 |
| 临时故障定位 | **eBPF** | — | 动态加载，无需重编 |
| 网络抓包分析 | **libpcap** | eBPF XDP | 生态成熟，tcpdump 兼容 |
| 进程生命周期监控 | **netlink connector** | eBPD | 零开销，精准事件流 |

## 六、实际项目的组合策略

> 这些技术不是替代关系，是互补关系。

以你的简历项目为例的三层防御：

```
eBPF（主力——覆盖面广，可编程）
  ├── syscall tracepoint → 文件/网络/进程行为审计
  ├── kprobe/fentry → 内核内部细节深挖
  └── cgroup/v2 → 容器级策略隔离

LD_PRELOAD（第一道防线——进程级精准拦截）
  └── 覆盖 libc 层的高危命令，查、杀延迟最低

fanotify（兜底文件防护网）
  └── 文件 open 前的同步决策，补 LD_PRELOAD 漏掉的路径

audit（合规兜底）
  └── 不可篡改日志，审计师认这个
```

## 七、本质抽象

| 技术 | 一句本质 |
|------|----------|
| eBPF | 在内核中运行**你的代码**，但在沙箱里 |
| LKM | 在内核中运行**你的代码**，没有沙箱 |
| LD_PRELOAD | 在**别人的进程**里运行你的代码 |
| inotify/fanotify | 订阅内核**预定义的文件事件** |
| audit | 订阅内核**预定义的 syscall 事件**，面向合规 |
| libpcap | 旁路复制**网络包**到用户态 |
| netlink | 订阅内核**子系统状态变更**的消息总线 |

> 再抽象一层：
> - **主动注入代码型**：eBPF、内核模块、LD_PRELOAD —— 你决定观测什么、怎么处理
> - **被动订阅事件型**：inotify/fanotify、audit、netlink —— 内核决定暴露什么，你只能选择订阅哪些
> - **旁路复制数据型**：libpcap —— 不关心逻辑，只复制原始数据

## 八、面试追问 · 考察候选人的四个层次

### 第一层 · 理解边界

> **问：** 你要监控生产环境所有容器的文件访问，记录谁（pid+容器）在什么时候打开了哪个文件。用哪个？为什么不用其他的？

**期望答案：**
- eBPF hook `sys_enter_openat`，内核态拿 pid + cgroup_id + filename
- 不用 audit：性能差，文本日志解析慢
- 不用 fanotify：没有容器信息，且要对每个挂载点单独设置
- 不用 LD_PRELOAD：Go 程序绕过、容器内部署复杂
- 不用 libpcap：这不是网络问题
- 不用内核模块：风险不可控，生产环境审批过不了

### 第二层 · 理解组合

> **问：** 如果你要做一个完整的容器安全观测系统，你会怎么组合这些技术？

**期望答案：**

```
eBPF（主力）
  ├── syscall tracepoint → 文件/网络/进程行为
  ├── kprobe/fentry → 内核内部函数细节
  ├── XDP/TC → 网络包级观测+策略执行
  └── cgroup/LSM → 容器级策略

Netlink PROC_CONNECTOR（辅助）
  └── 进程生命周期（fork/exec/exit 精确事件流）

fanotify（特定场景）
  └── 恶意文件扫描拦截（杀毒场景）

audit（合规兜底）
  └── 合规要求的不可篡改日志记录
```

### 第三层 · 理解取舍

> **问：** eBPF 这么好，为什么 audit 还没被淘汰？

**期望答案：**
- 合规标准（PCI-DSS、SOC2）明确要求 audit 日志，审计师只认这个
- audit 日志有**防篡改**语义（发到远端 syslog），eBPF ringbuf 不保证
- audit 有成熟的生态（ausearch、aureport、SIEM 集成）
- 部分场景不需要高性能，
