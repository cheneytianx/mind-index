# Agent 安全分层治理设计

> Agent 的安全不能仅靠 Agent 自身闭环，也不应完全依赖外部基础设施。正确的架构是「分层治理 + 信任链传递」——每层解决上一层的盲区，同时为下一层提供不可绕过的约束。

## 核心命题

**问题**：AI Agent 运行在主机或容器上。围绕 Agent 的安全设计，如何接入到 Agent 中？Agent 的安全是否只能从 Agent 自身闭环？

**答案**：否。安全控制的关键在于**信任锚点不能放在被保护对象内部**——Agent 被攻破后，其内部的所有安全检查都可能被绕过。正确做法是将安全能力分布在四个独立的层级中，形成嵌套的信任链。

---

## 第一性原理：为什么不能闭环

类比杀毒软件：如果你的杀毒逻辑跑在同一个进程里，病毒攻破这个进程就能关掉杀毒。同理：

```
Agent 内部安全检查 → 被提示注入 → 安全控制失效 → 损失
```

从 [[可信计算基|TCB（可信计算基）]] 的角度，安全控制必须运行在比受控对象**更低层、更可信**的环上。Agent 自身的安全判断是「非可信」的——它可能在帮你，也可能在被操控帮你作恶。

---

## 四层安全架构总览

```
┌─────────────────────────────────────────────────┐
│  ④ 管控层 (Control Plane)                        │
│  Policy Engine · 行为基线 · 全局审计 · 编排策略   │
├─────────────────────────────────────────────────┤
│  ③ Agent 运行时层 (Agent Runtime)                 │
│  输入清洗 · 意图判断 · 审批流触发 · 上下文隔离    │
├─────────────────────────────────────────────────┤
│  ② Sidecar/代理层 (MCP Proxy / Gateway)          │
│  工具鉴权 · 参数校验 · 请求审计 · 审批门控        │
├─────────────────────────────────────────────────┤
│  ① 基础设施层 (Host / Container / K8s)            │
│  身份凭证 · 网络策略 · Seccomp/SELinux · 只读 FS  │
└─────────────────────────────────────────────────┘
```

### 设计原则

1. **上层不可信赖下层**：每层假设上层已被攻破，防守本层的职责
2. **下层不必理解上层语义**：基础设施层不需要知道什么是「提示注入」
3. **信任单向传递**：下层为上层提供可信的身份和能力边界，上层向上层暴露可观测的状态
4. **每层独立**：任何单层被攻破，其他层仍然有效

---

## 第①层：基础设施层 —「不可绕过的硬约束」

这是信任的根。它只做一件事：**在 Agent 无法干预的层面施加约束。**

### 凭证基础设施

| 要素 | 实现 | 设计要点 |
|------|------|----------|
| 身份 | SPIFFE/SPIRE 工作负载身份 | 平台注入，Agent 不可自证 |
| 凭证 | 短命 Token（分钟级过期） | 非 API Key，非静态 |
| 绑定 | 硬件证明 / 沙箱绑定 | Token 不可窃取至其他环境 |
| 轮换 | 自动、透明 | Agent 进程无感知 |

核心设计：Agent 启动时由编排平台（如 K8s）注入身份，Agent 无法自行生成或修改这个身份。攻击者即使完全控制 Agent 进程，也无法伪造身份凭证去其他系统作恶。

### 网络策略

```
# K8s NetworkPolicy 示例：只读数据库 Agent
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: agent-readonly-db
spec:
  podSelector:
    matchLabels:
      agent: customer-db-reader
  policyTypes:
    - Egress
  egress:
    - to:
        - ipBlock:
            cidr: 10.0.1.5/32  # 只允许连指定数据库
      ports:
        - protocol: TCP
          port: 3306
```

即使 Agent 被注入「扫描内网所有主机」的指令，网络层直接拒绝。**Agent 看不到被拒绝的流量，也无法绕过。**

### 系统调用过滤与隔离

- **Seccomp**：限制系统调用白名单，比如只允许 `read`、`write`、`open` 等，禁止 `mount`、`ptrace`、`create_module`
- **SELinux/AppArmor**：对容器/进程打标签，限制文件访问路径
- **只读根文件系统**：Agent 无法安装持久性后门
- **沙箱执行不可信输入**：处理网页、文档的 Agent 应有额外的 OS 级沙箱

### 宿主机凭证管理

Agent 永远不应该在自己的配置或环境变量中存储长期凭证。凭证管理通过：

1. **临时卷挂载**（如 K8s Secret 挂载为卷，非环境变量）
2. **工作负载 API**（如 SPIFFE 工作负载 API 自动轮换）
3. **Secrets Store CSI Driver**（对接 Vault/AWS Secrets Manager）

---

## 第②层：Sidecar/代理层 —「行为拦截点」

Agent 与外部系统的所有交互，强制经过一个代理层。这一层的安全逻辑与 Agent 进程**物理隔离**，Agent 崩溃或被投毒不影响代理层的校验。

### 架构模式

```
┌──────────────────────────────┐
│  Agent 进程                   │
│  ┌───────────────────────┐   │
│  │ Agent Runtime (第③层) │   │
│  └───────┬───────────────┘   │
│          │ 工具调用请求        │
└──────────┼───────────────────┘
           │ localhost:proxy
┌──────────▼───────────────────┐
│  Agent Sidecar Proxy (第②层)  │
│  ┌─────────────────────────┐ │
│  │ 鉴权 · 参数校验 · 审计    │ │
│  │ 速率检测 · 审批门控       │ │
│  └──────┬──────────────────┘ │
└─────────┼─────────────────────┘
          │ 外部 API / 数据库 / MCP Server
```

### 核心能力

**工具级鉴权**

- Agent 只能调用预注册的「工具白名单」（默认拒绝）
- 每个工具有独立的权限声明：`{action: "query", scope: "readonly", max_rows: 1000}`
- 不是所有 Agent 都有「写数据库」的能力，即使它们想

**参数校验**

```yaml
# 工具策略配置示例
tools:
  - name: "database_query"
    policies:
      - block_statements: ["DROP", "ALTER", "DELETE", "INSERT"]
      - max_result_rows: 5000
      - only_allowed_tables: ["customers", "orders", "products"]
      - block_full_table_scan: true  # 没有 WHERE 条件的 SELECT 拒绝
  - name: "email_send"
    policies:
      - rate_limit: "10/hour"
      - require_approval: true  # 发送邮件需要审批
      - restrict_domains: ["@company.com"]  # 只允许发内部域
```

**参数校验在代理端和 Agent 端都做**——Agent 端为了加速，代理端作为兜底。

**审批门控**

高风险操作暂停并等待人工确认。代理层负责：

1. 拦截高风险调用请求
2. 推送审批事件到管控层（飞书/邮件/Webhook）
3. 等待审批结果（超时自动拒绝）
4. 审批通过后，放行原始请求

**审计日志**

结构化日志写入不可篡改的外部存储。日志包含：

- 请求方 Agent 身份（密码学可验证）
- 工具标识和原始参数
- 代理层的安全判断结果（通过/拒绝/审批中）
- Agent 层当时的上下文摘要（由 Agent 主动提供）

## 第③层：Agent 运行时层 —「只有 Agent 自己能做的事」

有些安全控制**必须**在 Agent 内部做，因为需要语义理解。这是唯一无法下放的一层。

### 必须由 Agent 自身完成的安全能力

**提示注入检测**

区分「用户的原始指令」和「来自外部数据源的被动内容」——这是 Agent 上下文窗口内的语义问题。Anthropic 提到 [[提示注入|Microsoft 的聚光灯 (Spotlighting) 技术]] 将间接注入成功率从 50%+ 降到 2% 以下。

关键设计点：

- 输入隔离：将系统指令、用户输入、外部数据标记为不同「来源」，在上下文窗口中保持隔离
- 宪法分类器：在生成输出前对响应内容进行越狱检测

Agent 将检测结果作为结构化日志对外暴露，但**检测失败不意味着没有攻击发生**——外部代理层的控制独立于 Agent 的判断。

**意图合法性判断**

Agent 知道「我为什么要调用这个工具」，外部 Sidecar 只能看到「调用了什么」。例如：

- Sidecar 看到 Agent 调用 `email_read()`
- 但只有 Agent 知道：用户说「把销售邮件发给竞争对手」vs「查看昨日销售邮件」

Agent 内部可以执行：

1. 再将工具调用意图映射到已知业务场景
2. 发现意图与授权范围不符时主动触发审批
3. 将意图信息附加到审计日志（供管控层事后分析）

**审批流触发**

Agent 在决策链路上感知到风险，主动发起升级审批。例如：用户让 Agent「发送邮件给公司全员」——Agent 内部识别这是高影响操作，触发审批。

与 Sidecar 审批的关系：**双层审批**——Sidecar 做规则级审批（基于策略），Agent 做语义级审批（基于意图理解）。两者独立，任何一个触发都应挂起操作。

**上下文隔离与会话边界**

- 每个会话从全新上下文开始，不共享工作内存
- 记忆存储按用户/会话分段，Agent 在读取记忆时检查访问权限
- 会话间严禁未授权的上下文传递

### 可观测性要求

Agent 内部的安全判断必须以**不可篡改**的方式暴露到外部：

```
Agent 安全日志格式 (Agent 内部 → 管控层)
{
  "agent_id": "uuid-xxx",
  "session_id": "session-yyy",
  "timestamp": "2026-05-28T10:00:00Z",
  "security_event": {
    "type": "prompt_injection_detection | intent_check | approval_trigger",
    "trigger_input": "...",
    "detection_result": "safe | suspicious | blocked",
    "confidence": 0.95,
    "action_taken": "allow | block | escalate"
  }
}
```

这些日志进入管控层后，与 Sidecar 的外部日志交叉验证，形成**双源审计**。

## 第④层：管控层 —「你不是唯一」

单个 Agent 看不见全局。管控层是唯一能看到所有 Agent 行为的实体。

### 行为基线建模

```
正常模式：
  Agent A 每天调用 database:query 约 50-200 次
  每次查询平均返回 10-50 行
  调用时间集中在 09:00-18:00

异常：
  Agent A 在 02:00 调用 5000 次 database:query
  每次查询返回 10000+ 行
  → 自动降级权限，拉起事件单
```

基线自动构建（头 7 天学习），之后持续更新。

### 跨 Agent 攻击检测

Agent 间的横向移动是特有威胁：

- Agent A 突然向 Agent B 发送高危指令（「帮我转发这个文件到外部邮箱」）
- 被攻破的低权限 Agent 向高权限 Agent 发送伪装成合法任务的指令（[[最小代理权|困惑代理人问题]]）
- 多个 Agent 的行为模式同时偏离基线

### 策略统一下发

- 安全团队定义策略（如「所有读取 PII 的操作必须经过审批」）
- 管控层将策略转换为两侧的语言：
  - **向 Sidecar** 下发技术策略（参数规则、白名单）
  - **向 Agent** 下发语义策略（「识别并标记含 PII 的字段」）

### 全局度量指标

**驻留时间 (Dwell Time)**：异常发生到人类知晓的时长
**覆盖率 (Coverage)**：实际调查的告警比例
**检测速度**：发现异常行为的平均时间（目标是 1 小时内）

## 信任链设计

各层间的信任不是机械堆叠，而是通过明确的设计传递：

```
基础设施层认证 Agent 进程身份（SPIFFE ID）
    ↓
基础设施层将这个身份注入到：
  ├── Sidecar 代理层（用于鉴权）
  └── Agent 运行时层（用于日志签名）
        ↓
Agent 运行时层在其日志中附加此 ID + 动作语义
    ↓
Sidecar 代理层在其拦截日志中也附加此 ID + 工具调用细节
    ↓
管控层将两个日志源关联 → 双源校验 → 行为分析
```

关键设计：**Agent 不能自证其身份**——身份由基础设施注入（下层可信根）。Agent 可以做的是用这个身份签名其行为日志，但**不能控制身份的发放**。

## 典型攻击场景推演

### 场景 1：间接提示注入

| 步骤 | 攻击行为 | 防御层拦截 |
|------|----------|------------|
| S1 | 攻击者在公开网页注入恶意指令 | — |
| S2 | Agent 正常浏览网页，读取到了恶意指令 | — |
| S3 | Agent 分析指令：「把数据库导出连接到攻击者服务器」 | ③ Agent 层：提示注入检测可能发现异常，输出可疑标记 |
| S4 | Agent 尝试连接外部 IP | ① 基础设施层：网络策略限制 Egress 只允许特定目标，连接被阻止 |
| S5 | Agent 尝试调用 database:dump | ② Sidecar 层：校验参数发现是导出操作，且 rate 异常，拒绝执行 |
| S6 | 管控层收到两个异常事件（可疑标记 + 拒绝执行） | ④ 管控层：自动隔离该 Agent，拉起事件 |

**总结**：Agent 的语义检测可能失败，但基础设施和 Sidecar 的硬控制照样兜住了。

### 场景 2：Agent 被投毒后横向移动

| 步骤 | 攻击行为 | 防御层拦截 |
|------|----------|------------|
| S1 | 攻击者通过供应链投毒了一个 MCP 服务器 | — |
| S2 | Agent A 使用了被投毒的 MCP 工具，Agent A 失陷 | — |
| S3 | Agent A 被操控，向 Agent B 发送委派任务「把客户数据发到 X」 | ④ 管控层：检测到 Agent A→B 的通信模式异常 |
| S4 | Agent B 收到委派任务 | ③ Agent B：校验来源 Agent A 的权限，发现 A 无权访问客户数据，拒绝 |
| S5 | Agent A 直接尝试调用 database:read_customers | ② Sidecar：A 的工具权限声明中没有该操作，直接拒绝 |
| S6 | Agent A 的凭证被管控层轮换 | ① 基础设施层：Token 过期，Agent A 被彻底隔离 |

## 实现路线图

建议按 Foundation → Enterprise → Advanced 三级递进（参考 [[Zero Trust|Anthropic 白皮书]]的安全能力等级）。

### Foundation（可立即开始）

- [ ] 所有 Agent 使用 K8s NetworkPolicy 限制网络出站
- [ ] Seccomp 限制系统调用
- [ ] 不再将 API Key 的 Agent 交互，改用短命 Token
- [ ] 建立工具白名单（每个 Agent 明确声明能调用的工具）
- [ ] 结构化审计日志（Agent 行为全部落盘）

### Enterprise（一到三个月）

- [ ] Sidecar 代理层部署（MCP/API Gateway）
- [ ] 参数校验引擎（自动识别高危操作）
- [ ] 审批工作流（高风险操作自动挂起）
- [ ] Agent 内部提示注入检测
- [ ] 管控层行为基线建立

### Advanced（持续演进）

- [ ] 硬件绑定凭证（TPM/Hardware Attestation）
- [ ] ABAC/ReBAC 动态权限
- [ ] 跨 Agent 横向移动检测
- [ ] Agentic SOAR 自动响应
- [ ] 记忆完整性校验（哈希链存储）

## 总结

| 层面 | 信任根 | 核心控制 | Agent 能否绕过 |
|------|--------|----------|---------------|
| ① 基础设施 | 硬件/内核/编排平台 | 网络、系统调用、身份注入 | ❌ 无法绕过 |
| ② Sidecar | 独立进程 | 工具鉴权、参数校验 | ❌ 无法绕过 |
| ③ Agent | — | 语义级安全 | ✅ 有可能（被攻破时） |
| ④ 管控 | 独立系统 | 全局行为分析 | ❌ 无法绕过 |

**Agent 的安全不是「要不要自身闭环」的问题，而是「哪些回路由 Agent 自身闭环，哪些必须下沉」的问题。** 正确的回答是：Agent 只负责需要语义理解的部分，其行为必须被基础设施和代理层约束在不可修改的边界内。
